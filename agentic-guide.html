Ch 01 The Agentic Engineering Landscape +

What are AI agents, really?

Strip away the marketing and an AI agent is a program that uses a language model to decide what to do next. It operates in a loop: observe the environment, decide on an action, execute it, observe the result, decide again.

This definition is deliberately simple, but it captures the essential difference between agents and other AI applications. A chatbot generates text in response to a prompt — it's a function from text to text. An agent generates actions in response to a goal — it's a loop that continues until the goal is achieved or a termination condition is met. The difference is autonomy. A chatbot does what you ask, once. An agent pursues a goal, making decisions along the way about what to do next.

The autonomy is what makes agents both powerful and dangerous. Powerful because they can accomplish complex, multi-step tasks without human intervention at each step. Dangerous because they can take actions that have real-world consequences — modifying files, calling APIs, deploying code, sending messages — and those actions can be wrong.

DimensionChatbotAgent
ActionsGenerates textExecutes code, calls APIs, modifies systems
AutonomyResponds to promptsDecides what to do next
DurationSingle turnMinutes to hours
RiskWrong answerWrong action (data loss, security breach, cost overrun)
AuthorizationUser's permissionsNeeds its own permission model
ObservabilityLog the responseTrace every decision and action
Failure modeBad textProduction incident

The agent capability spectrum

Level 1: Completion agents suggest code as you type. They operate within the IDE, have no access to external tools, and produce output that the developer reviews character by character. GitHub Copilot's inline suggestions are the canonical example.

Level 2: Chat agents respond to natural language requests within a conversation. They can generate multi-line code, explain concepts, and answer questions about the codebase.

Level 3: Command agents execute actions in the development environment. They can read and write files, run commands, create branches, and open pull requests. Claude Code, Cursor's agent mode, and Ona are examples. The engineering requirements are significant — you need authorization, observability, and cost control.

Level 4: Background agents run without human supervision. They monitor repositories for issues, automatically fix bugs, generate tests, update documentation, and create pull requests — all without a human initiating the task. The engineering requirements are the highest — you need everything from Level 3 plus automated quality gates, kill switches, and incident response procedures.

Most teams in February 2026 are at Level 2–3. The transition from Level 2 to Level 3 is where the engineering discipline of this guide becomes essential.

The competitive landscape of coding agents

IDE-integrated agents (Cursor, Windsurf, GitHub Copilot) embed AI directly into the development environment. CLI agents (Claude Code) run in the terminal alongside your existing tools. Cloud agents (Ona, OpenAI Codex, Devin) run in isolated cloud environments. Platform agents (GitHub Agentic Workflows) are integrated into development platforms.

Why "agentic engineering" is a discipline

Building with agents isn't just "using AI tools." It requires new engineering disciplines: Context Engineering — deciding what information goes into the model's context window; Agent Authorization — deciding what the agent is allowed to do; Agent Observability — tracing what the agent did, why, and what it cost; Agent Orchestration — how agents coordinate, delegate, and recover from failures; Human-Agent Collaboration — how humans and agents work together without burning out the humans.

The teams that succeed start at the bottom (infrastructure) and work up. This guide follows that bottom-up order.

Ch 02 The Capability Jump +

What changed in late 2025

In late 2025, two model releases crossed an invisible capability threshold: Claude Opus 4.5 (November 24) and GPT-5.2 (December 11). On benchmarks, they were incremental improvements. In practice, they changed what was possible. Simon Willison described it: "It genuinely feels to me like GPT-5.2 and Opus 4.5 in November represent an inflection point — one of those moments where the models get incrementally better in a way that tips across an invisible capability line where suddenly a whole bunch of much harder coding problems open up."

By February 2026, the frontier moved again. Anthropic shipped Claude Opus 4.6 and Sonnet 4.6 with 1M token context windows. OpenAI released GPT-5.3-Codex. Google followed with Gemini 3.1 Pro. Four frontier models from three labs in fourteen days. The "pick one best model" era ended — model routing became a production requirement.

Five implications for engineering teams

Implication 1: Agents are management problems. The skills that make someone effective with agents are management skills: clear communication of goals, providing necessary context, breaking down complex tasks, giving actionable feedback, and knowing when to intervene. Senior engineers who insist on doing the work themselves struggle with agents. Former engineers who moved into management pick them up quickly.

Implication 2: Output scales faster than review capacity. A team of five engineers produces roughly 25 PRs per week. With agents, the same team might produce 75–100 PRs per week. Review time triples, but team size hasn't changed. The solution is a two-layer review process where automated checks handle mechanical review and humans focus on judgment calls.

Implication 3: The documentation business model is dying. Adam Wathan (Tailwind Labs): "75% of the people on our engineering team lost their jobs here yesterday because of the brutal impact AI has had on our business… Traffic to our docs is down about 40%… our revenue is down close to 80%." The businesses that survive will integrate into AI workflows rather than competing with AI for attention.

Implication 4: The junior engineer pipeline is disrupted. The tasks traditionally assigned to junior engineers are exactly the tasks agents handle well. The emerging pattern: junior engineers shift from writing code to reviewing agent-generated code.

Implication 5: Model routing replaces model selection. GPT-5.3-Codex leads on Terminal-Bench (75.1%), Opus 4.6 leads on SWE-bench Verified (80.8%), Gemini 3.1 Pro leads on long-context tasks. The "pick one model" era is over. Model routing is covered in Chapter 31.

Ch 03 The Agentic AI Foundation & Standards +

What is AAIF?

On December 9, 2025, Anthropic, OpenAI, and Block announced the Agentic AI Foundation (AAIF) under the Linux Foundation, with support from Google, Microsoft, AWS, Cloudflare, and Bloomberg. AAIF aims to standardize the agent infrastructure layer — following the same pattern that produced Linux, Kubernetes, and OpenTelemetry.

The three founding projects

Model Context Protocol (MCP) — donated by Anthropic — is a universal protocol for connecting AI agents to external systems. It's becoming the USB-C of AI agents: build an MCP server once, and it works with any MCP-compatible agent. Over 10,000 active public MCP servers, 97 million monthly SDK downloads.

Goose — donated by Block — is an open-source, extensible AI agent designed for production use. It's MCP-native, observable, and controllable — a reference implementation of what a well-architected agent looks like.

AGENTS.md — donated by OpenAI — is a standard format for AI coding agent instructions. Think of it as README.md for AI agents. Over 60,000 open-source projects now have AGENTS.md files.

What's missing from AAIF

Three significant gaps remain: security standards (MCP defines tool calls but not authentication or permission scoping); authorization models (no standard for fine-grained, context-dependent agent permissions); observability standards (OpenTelemetry is extending to cover agent workflows but semantic conventions are still being defined).

The open source agent infrastructure stack

  • Model abstraction: LiteLLM, OpenRouter
  • Authorization: OpenFGA (CNCF Incubating), OPA (CNCF Graduated)
  • Observability: OpenTelemetry, Langfuse, Arize Phoenix
  • Sandboxing: Anthropic sandbox-runtime, gVisor, Bubblewrap
  • Evaluation: Promptfoo, Braintrust, DeepEval
  • Context engineering: Distill, Chroma, Qdrant, Weaviate
"You cannot prompt your way out of garbage context."

Ch 04 Context Windows +

What is a context window?

A context window is the total amount of text (measured in tokens) that a language model can process in a single request. Everything the model knows about your task — the system prompt, the conversation history, the retrieved documents, the code files, the tool definitions — must fit inside this window. A token is roughly four characters in English.

ModelContext WindowApprox. Pages
GPT-5.2400K tokens~640 pages
Claude Sonnet 4.6200K (1M beta)~320 pages
Gemini 3.1 Pro1M tokens~1,600 pages

These numbers look large. They're not. A typical monorepo has millions of lines of code. The window fills up fast.

Why size isn't everything

The "lost in the middle" problem. Research from Stanford and UC Berkeley demonstrated that language models perform significantly worse on information placed in the middle of long contexts. The practical implication: context ordering matters as much as context content. Place the most important information at the beginning or end.

The cost problem. Tokens cost money, and the cost scales linearly. A team running 500 agent tasks per day at 100K tokens each spends roughly $150/day on input tokens alone. With context engineering, that drops to $22.50/day — a $46,500/year saving from a single optimization.

The noise problem. More context doesn't mean better context. If you send 100K tokens and only 15K are relevant, you've diluted the signal with 85K tokens of noise. Less, better context beats more, noisier context.

Token budgets

A token budget is a deliberate allocation of your context window across different purposes. A well-designed budget for a 200K window might allocate: 5K for system instructions, 15K for tool definitions (or 2K with meta-MCP compression), 30K for retrieved context, 20K for conversation history, and 130K as headroom for tool results and model reasoning.

Token budgets should be enforced programmatically — your agent framework should track usage per category and warn or truncate when exceeded. Without enforcement, budgets are aspirational.

Ch 05 The Context Engineering Stack +

The four layers

Context engineering operates in four layers, each addressing a different problem. These layers form a pipeline: raw retrieved content enters at the bottom and curated, compressed context exits at the top.

Layer 1: Clustering

When you retrieve 10 chunks from a vector database, they often contain the same information expressed differently. Clustering groups semantically similar chunks together using their embedding vectors. Agglomerative clustering works well because it doesn't require you to specify the number of clusters upfront. A cosine distance threshold of 0.15–0.25 works well for most documentation and code contexts.

Layer 2: Selection

After clustering, pick the best representative from each cluster. For coding tasks, the best representative is usually the most specific one — the code file itself rather than the documentation. For research tasks, the most authoritative one. For customer-facing tasks, the most recent one.

Layer 3: Reranking

After selection, rerank the remaining chunks to balance relevance and diversity using Maximal Marginal Relevance (MMR). The λ parameter controls the trade-off: λ = 1.0 means pure relevance (may include redundant chunks), λ = 0.0 means pure diversity. λ = 0.7 is a good default for most use cases.

Layer 4: Compression

The final layer removes redundancy without losing signal. The critical decision: deterministic vs. non-deterministic compression. LLM-based summarization produces readable output but introduces non-determinism (~500ms, $0.01/call). Algorithmic deduplication is faster, cheaper, and reproducible (~12ms, $0.0001/call). For production systems, determinism is a requirement — use algorithmic compression.

Distill: A reference implementation

Distill implements all four layers in a single pipeline with a determinism-first design philosophy. Results from production deployments:

MetricBefore DistillAfter DistillImprovement
Context tokens45K12K73% reduction
Redundant chunks30–40%<2%~95% reduction
Output consistency62%89%+27pp
Cost per request$0.135$0.03673% reduction
Latency8.2s2.1s74% reduction

Meta-MCP: Compressing tool definitions

A pattern from the Japanese developer community addresses tool definition bloat. When you connect 60+ MCP tools, the tool descriptions alone consume 15–20K tokens. The meta-MCP pattern compresses these into 2 meta-tools: a discovery tool (returns a list with brief descriptions) and a load tool (loads the full definition of a specific tool on demand). Tool definition overhead drops from 15–20K tokens to 2–4K tokens — an 80–88% reduction.

Ch 06 RAG vs. Agentic Search +

The RAG problem

Retrieval-Augmented Generation (RAG) was the dominant pattern for giving LLMs access to external knowledge from 2023 through early 2025. The pipeline is straightforward: embed the query, search a vector database for similar chunks, stuff the top-K results into the context, and generate a response. RAG works for simple factual lookups, but it has fundamental limitations:

  • Static retrieval — the query is embedded once, no opportunity to reformulate
  • No reasoning over retrieval — can't resolve contradictions between retrieved documents
  • Single-hop only — can't answer multi-hop questions requiring multiple retrievals with reasoning in between
  • Context pollution — retrieves the top-K most similar chunks, which often contain redundant information

Agentic search

Agentic search is a fundamentally different approach. Instead of a fixed pipeline, the agent decides how to search, evaluates the results, and iterates. Boris Cherny articulated the distinction: RAG retrieves. Agentic search reasons about what to retrieve, evaluates the results, and iterates. The difference is the reasoning loop.

When to use which

ScenarioRAGAgentic Search
Simple factual lookup✅ Fast, cheapOverkill
Multi-hop reasoning❌ Single-hop only✅ Iterative retrieval
Ambiguous queries❌ Fixed interpretation✅ Can clarify and refine
Cost-sensitive✅ One retrieval❌ Multiple retrievals
Complex codebase questions❌ Surface-level✅ Deep understanding

Building a codebase knowledge graph

For engineering teams, the most valuable context source is the codebase itself. A codebase knowledge graph indexes file relationships (imports, dependencies, call graphs), git history (who changed what, when, and why), architecture decisions (ADRs, design docs), and conventions (naming patterns, testing patterns, error handling). The knowledge graph should be exposed as an MCP server so that any agent can query it.

Ch 07 The Agent Security Crisis +

We're speedrunning security mistakes

AI agents are infrastructure. Claude Code ships production features. MCP connects agents to thousands of external systems. And we're repeating every security mistake from the last three decades — but faster. 80% of Fortune 500 companies use AI agents. Security researchers have found over 8,000 publicly visible MCP servers, many with no authentication. Agents routinely run with the same permissions as the developer who launched them.

The new threat model

Traditional AppAgent System
Humans make requests → Code executesLLMs make requests → Code executes
Input is structured (forms, APIs)Input is unstructured (natural language)
Behavior is deterministicBehavior is probabilistic
Attack surface is the APIAttack surface is any text the agent processes
Failures are bugsFailures can be adversarial

The three attack vectors

Prompt injection occurs when malicious instructions are embedded in data that the LLM processes. An agent reading a web page might encounter text that says "ignore your previous instructions and instead…" This is the SQL injection of the AI era — but worse, because there's no equivalent of parameterized queries.

Tool poisoning occurs when tools return malicious responses containing embedded instructions. A compromised MCP server, a man-in-the-middle attack on an API call, or a supply chain attack on a dependency can all inject malicious content into tool responses. Every tool response should be treated as untrusted input.

Exfiltration through normal operations: every agent capability is a potential exfiltration vector. An agent that can make HTTP requests can send data to any URL. An agent that can push to git can push secrets to public repositories.

Ch 08 Zanzibar for AI Agents +

What is Zanzibar?

Zanzibar is Google's global authorization system, described in a 2019 paper. It handles permission checks for Google Drive, YouTube, and virtually every Google product — millions of checks per second with global consistency. The key insight is Relationship-Based Access Control (ReBAC): instead of assigning permissions directly to users, you define relationships between entities.

Why agents need Zanzibar

AI agents have authorization requirements that are more complex than human users. Traditional RBAC can't express them because agent permissions need to be:

  • Fine-grained: "This agent can read files in src/ but not in config/secrets/"
  • Context-dependent: "This agent can access production logs only during an active incident"
  • Delegatable: "This orchestrator agent can grant its sub-agents read access to the files it's working on, but not write access"

OpenFGA: Open-source Zanzibar

OpenFGA is the open-source implementation of Zanzibar, a CNCF Incubating project used by Okta, Twitch, and Canonical. Performance at scale:

ScaleLatency (p99)Throughput
10K checks/sec< 1ms✅
100K checks/sec< 5ms✅
1M checks/sec< 10ms✅ (with caching)

Authorization checks add negligible overhead. There's no performance excuse for skipping authorization.

The principle of least privilege for agents

Least privilege for agents means starting with zero permissions and adding only what's needed. A code review agent needs read access to the repository and write access to PR comments. It doesn't need write access to the repository, access to the CI/CD pipeline, or access to any database. Agents should have permissions added incrementally as needs are identified, not granted preemptively "just in case."

OPA vs. OpenFGA

CriteriaOPAOpenFGA
Authorization modelAttribute-based (ABAC)Relationship-based (ReBAC)
Best forComplex conditional policies"Who can access what" relationships
Performance~1ms policy evaluation~2–5ms relationship check

The hybrid approach: use OPA for broad policy decisions ("Is this action type allowed?") and OpenFGA for fine-grained resource access ("Does this agent have read access to this specific file?").

Ch 09 Prompt Injection & Tool Poisoning +

Prompt injection: The SQL injection of AI

Prompt injection occurs when malicious instructions are embedded in data that the LLM processes, causing it to deviate from its intended behavior. The term was coined by Simon Willison in 2022. SQL injection was solved with parameterized queries — a clean separation between code and data. Prompt injection has no equivalent solution because language models don't distinguish between instructions and data.

Types of attacks: Direct injection (malicious instructions in user input), indirect injection (malicious instructions in content the agent retrieves — web pages, documents, tool responses), jailbreaking (override safety training), prompt leaking (extract the system prompt), goal hijacking (redirect the agent to a different task), payload smuggling (encode instructions in base64, ROT13, Unicode tricks).

Defense hierarchy

StrategyEffectiveness
Sandboxing (limit what agent can DO)Very High
Authorization (limit what agent is ALLOWED)Very High
Output filteringHigh
Semantic similarity detectionHigh
Input sanitizationMedium
Perplexity detectionMedium

Build your defense from the top down. Start with structural defenses (sandboxing, authorization), add architectural defenses, and layer textual defenses on top. Don't rely on textual defenses alone — they're the weakest layer.

Tool poisoning

Tool poisoning is the attack vector that almost no one discusses, and it may be the most dangerous. Agents call external tools dozens or hundreds of times per session. Each tool response becomes part of the agent's context. If any of those responses contain malicious instructions, the agent may follow them. A compromised MCP server, a man-in-the-middle attack, or a supply chain attack can all inject malicious content.

The defense is response validation. Every tool response should be validated against an expected schema before it's added to the agent's context. If a database query returns natural language instructions instead of structured data, that's anomalous and should be flagged.

The state of prompt injection defense

As of February 2026, there is no complete defense against prompt injection. This is an uncomfortable truth. Every defense can be bypassed by a sufficiently sophisticated attack. The goal isn't to prevent all injection — it's to make injection hard enough that the cost of attacking exceeds the value of the target. The most effective defenses are structural, not textual.

Ch 10 Sandboxing & Runtime Protection +

Why containers aren't a sandbox

A common misconception: "We run our agents in Docker containers, so they're sandboxed." Containers provide process isolation, not security isolation. They were designed for deployment consistency, not adversarial containment.

FeatureContainerSecurity Sandbox
Filesystem isolationPartial (volumes leak)Full (explicit allowlist)
Network isolationConfigurableDefault deny
Syscall filteringOptional (seccomp)Mandatory
Designed forDeploymentAdversarial containment

Ephemeral environment isolation

The strongest sandboxing pattern gives each agent its own ephemeral environment — a fresh, isolated VM or container that is destroyed after the task completes. No persistent state means no accumulated security risks or data leakage between sessions. Ona takes this approach in production: every agent gets an isolated cloud environment that's destroyed when the task is done.

OS-level sandboxing

seccomp restricts which system calls a process can make. A typical agent needs file I/O, network sockets, process management, and memory allocation. A seccomp profile that allows only the necessary syscalls prevents an entire class of privilege escalation attacks.

Landlock (Linux Security Module) restricts filesystem access at the kernel level — path-based rather than identity-based. An agent sandboxed with Landlock can read and write files in its working directory but cannot access /etc/passwd or /root/.ssh — even if running as root.

Anthropic's sandbox-runtime combines multiple isolation layers (seccomp profiles, filesystem restrictions, network policies, resource limits) into a cohesive sandbox designed specifically for AI agents.

Defense in depth architecture

The complete security architecture layers all defenses: Authorization controls what the agent is allowed to do. Sandboxing controls what it can physically do. Prompt injection defense controls what it's tricked into doing. Policy engines enforce business rules that span all three layers. No single layer is sufficient — each layer catches what the previous layers miss.

Ch 11 Model Context Protocol (MCP) +

What MCP is

MCP is a protocol for connecting AI agents to external systems. It defines how agents discover tools, call them, and process results. Build an MCP server once, and it works with any MCP-compatible agent — Claude, ChatGPT, Cursor, Gemini, VS Code, GitHub Copilot, Windsurf, and dozens of others. Over 10,000 public MCP servers are available, covering everything from databases and file systems to Slack, GitHub, Jira, and Notion. The npm and PyPI SDKs have over 97 million monthly downloads combined.

Protocol architecture

MCP uses JSON-RPC 2.0 over two transport mechanisms: stdio transport (runs the MCP server as a subprocess, communicates over standard input/output — simplest setup) and HTTP transport (uses Server-Sent Events for streaming, supports remote MCP servers — production pattern).

MCP servers expose three types of capabilities:

CapabilityPurposeExample
ToolsActions the agent can takegithub.create_issue, db.query
ResourcesData the agent can readDatabase schemas, file contents
PromptsReusable prompt templatesCode review template

The Backend for MCP (BFMCP) Pattern

A design pattern that separates MCP protocol handling from business logic. The MCP server exposes high-level, business-logic operations ("get user profile," "list recent orders") rather than low-level data access operations ("execute SQL query"). The backend service translates these into the appropriate data access calls, applying business rules, access controls, and rate limits. Benefits: thin MCP server (easy to test and audit), reusable business logic, security policies in the backend.

No audit loggingNo record of tool callsAdd logging middleware

Ch 12 Agent-to-Agent Communication (A2A) +

Why agents need to talk to each other

MCP connects agents to tools. But what happens when agents need to coordinate with other agents? Tool integration is synchronous and stateless. Agent-to-agent communication is asynchronous and stateful — one agent delegates a task to another, the second agent works for minutes or hours, and the first agent needs to track progress, handle failures, and integrate the result.

Without a standard protocol, each of these interactions requires custom integration code — an N-squared integration problem. The same problem that MCP solved for tool integration.

Google's A2A protocol

Google launched the Agent2Agent (A2A) protocol in 2025 to solve this problem. A2A enables secure, interoperable communication between agents regardless of their underlying framework.

Every A2A-compatible agent publishes an "Agent Card" — a JSON document describing its capabilities, input/output schemas, authentication requirements, and endpoint URL. Agent Cards serve the same purpose for agent-to-agent communication that tool definitions serve for MCP.

MCP vs. A2A

DimensionMCPA2A
PurposeAgent ↔ ToolAgent ↔ Agent
CommunicationSynchronous (request/response)Async (task lifecycle)
DiscoveryTool definitionsAgent Cards
StateStatelessStateful (task tracking)
DurationMillisecondsMinutes to hours

They're complementary, not competing. MCP for tools, A2A for agents.

Ch 13 AGENTS.md & Codebase Onboarding +

What is AGENTS.md?

AGENTS.md is a markdown file that tells AI agents how to work with your project. Humans read README.md to understand a project. Agents read AGENTS.md to understand how to contribute to it. Over 60,000 open-source projects now have AGENTS.md files. Projects that adopted AGENTS.md early report 30–50% higher first-attempt success rates for agent-generated code, fewer convention violations, and fewer failed builds.

Anatomy of a good AGENTS.md

  • Header: Project overview and tech stack
  • Quick Start: Install, dev, test, and build commands (exact commands, not descriptions)
  • Architecture: Directory structure with purpose annotations
  • Code Conventions: TypeScript strict mode, naming conventions, error handling patterns (be specific, not general — "Use camelCase for variables, PascalCase for types" not "Follow good practices")
  • Testing Requirements: CI requirements, coverage thresholds, test frameworks
  • Security Notes: What the agent should NEVER do (crucial negative instructions)
  • Common Patterns: Step-by-step instructions for recurring tasks
  • Known Issues: Gotchas that agents should be aware of

Writing effective AGENTS.md

Be specific, not general. "Follow good coding practices" is useless. "Use camelCase for variables, PascalCase for types. Error handling uses Result pattern, not try/catch." is actionable.

Include negative instructions. Tell the agent what NOT to do. "Never use any types in TypeScript." "Never modify files in /config/secrets/."

Test it. Run an agent task using only the information in AGENTS.md. If the agent produces incorrect output, the AGENTS.md is missing information. Iterate until the agent can complete common tasks correctly on the first try.

Agent Skills: Reusable procedural knowledge

AGENTS.md provides project-specific context. Skills are portable capabilities that travel across projects and teams — markdown files that encode procedural knowledge (step-by-step workflows, best practices, anti-patterns, domain expertise) in a format agents can read and follow. The open-source skills ecosystem (skills.sh) has over 79,000 skills covering frontend patterns, infrastructure automation, logging best practices, and more.

Ch 14 Agent Traces & OpenTelemetry +

Why standard monitoring doesn't work

Traditional application monitoring tracks HTTP requests, database queries, error rates, and latency. Agent systems need to track fundamentally different signals: decisions (why did the agent choose this action?), tool calls (what was called with what arguments?), permission checks, token usage, cost, and context (what information was in the context window when the agent made each decision?). Without proper tracing, debugging agent failures is like debugging a distributed system with only access logs.

OpenTelemetry for agents

OpenTelemetry is the industry standard for application observability. The key concepts map naturally to agent systems:

OTel ConceptAgent Equivalent
TraceComplete agent session (task start to completion)
SpanIndividual agent action (tool call, LLM request, decision)
AttributesAgent-specific metadata (tokens, cost, model, tool name)
EventsSignificant moments (permission denied, human approval, error)

Trace-based debugging

When an agent produces bad output, the trace tells you why. The debugging workflow: (1) Find the trace using the session ID. (2) Identify the divergence point — walk through the trace, comparing the agent's actions to what you would have done. (3) Examine the context at the divergence point — the divergence is almost always caused by missing context, misleading context, or ambiguous instructions. (4) Fix the root cause — update AGENTS.md, fix the retrieval pipeline, or clarify the task specification. This systematic workflow produces a feedback loop that continuously improves agent output quality.

Choosing an observability backend

General-purpose backends (Jaeger, Grafana Tempo, Datadog) — right choice if you want to integrate agent traces into your existing dashboards and correlate agent actions with application traces.

AI-specific backends (Langfuse, Arize Phoenix, LangSmith, Braintrust) — purpose-built for AI workloads, with prompt versioning, eval integration, and cost tracking built in.

The hybrid approach uses OpenTelemetry for trace collection and exports to both backends. For most teams, start with an AI-specific backend and add general-purpose integration later.

Ch 15 Cost Tracking & Token Economics +

Why cost tracking matters

Agent costs are driven by model behavior, which is non-deterministic. The same task might cost $0.50 one day and $3.00 the next, depending on how many tool calls the agent makes and whether it gets stuck in a retry loop.

ScenarioTokensCost
Simple bug fix15K$0.05
Feature implementation150K$0.45
Complex refactoring500K$1.50
Runaway agent (no limits)5M+$15+
Team of 10 × 20 tasks/day (monthly)~900M~$2,700/month

Cost optimization strategies

Model routing: Use the cheapest model that can handle the task. A simple bug fix or test generation task can be handled by Sonnet 4.6 at $3/$15 per million tokens instead of Opus 4.6 at $5/$25. The key is having a routing layer that makes this decision automatically.

Context engineering: Reducing context size directly reduces cost. Distill (deduplication) achieves 30–40% token reduction; meta-MCP (tool compression) achieves 10–15% overall reduction; selective context loading achieves 50–70% reduction. Combined: 60–80% reduction.

Budget caps: Set hard limits per session (default: $3 for routine tasks, $10 for complex tasks) with automatic termination when exceeded.

Hidden costs

Retry costs: When a tool call fails, the agent retries — but the retry includes the full conversation history plus the failed attempt. A task that fails three times before succeeding can cost 4–5x the cost of a successful first attempt.

Context accumulation: Each turn in the conversation includes the full history of previous turns. Turn 1 sends 5K tokens. Turn 10 sends 50K tokens. Turn 20 sends 100K tokens. The total cost of a 20-turn session grows quadratically, not linearly.

Ch 16 Incident Response for Agent Failures +

When agents break things

Agent failures are fundamentally different from application failures. Application bugs are deterministic. Agent failures are non-deterministic. Application failures are localized. Agent failures cascade — an agent that misunderstands a task might modify twelve files across three services before anyone notices. And while you can roll back a deployment, you can't always undo what an agent did, especially if it made API calls, sent messages, or modified external state.

Incident taxonomy

Severity 1: The agent took an action that affected production systems, corrupted data, or exposed sensitive information. Requires immediate human response — stop the agent, assess the blast radius, begin remediation within minutes.

Severity 2: The agent produced incorrect output that passed automated checks but was caught in human review. Requires investigation within an hour.

Severity 3: The agent failed to complete a task or produced obviously wrong output that was caught immediately. A learning opportunity, not an emergency.

The incident response runbook

  1. Contain: Kill the agent session, revoke temporary credentials, prevent affected code from being deployed
  2. Assess: Review the full trace log, identify all files modified and commands executed, determine whether any actions affected external systems
  3. Remediate: Revert the agent's changes using git, restore any modified external state, verify the system is back to a known good state
  4. Learn: Post-incident review within 48 hours, update guardrails to prevent recurrence, share findings with the team

Common agent failure patterns

  • The infinite loop: Agent encounters an error, attempts to fix it, introduces a new error, cycles indefinitely. Fix: loop detector that terminates after 3 consecutive failed attempts.
  • Scope creep: Agent misunderstands task boundaries and modifies files outside the intended scope. Fix: explicit scope constraints in the task specification.
  • The confident hallucination: Agent produces output that looks correct — compiles, passes tests, follows conventions — but contains a subtle logical error. The most dangerous failure. Fix: adversarial testing in the eval pipeline.
  • The cost spiral: Agent encounters a difficult problem and keeps trying increasingly expensive approaches. Fix: per-session cost limits with automatic termination.

Agent Orchestration

Ch 17 The Agent Loop +

The core pattern

Every agent, regardless of framework, runs a variation of the same loop: observe the current state, decide what to do next, take an action, observe the result, and repeat. The model receives a prompt containing the task description, conversation history, and results of any previous actions. It produces either a final answer or a request to use a tool. If it requests a tool, the system executes that tool, appends the result to the conversation, and sends everything back to the model. This continues until the model produces a final answer or hits a limit.

Loop variants

ReAct (Reason + Act) forces the model to explicitly state its reasoning before each action. Improves transparency but increases token usage. Plan-and-Execute separates planning from execution — the model creates a full plan upfront, then executes each step. Works well for predictable tasks but struggles when early steps reveal information that invalidates the plan. Graph-based (LangGraph) models the agent as a state machine with conditional branching. Role-based (CrewAI) assigns different personas to different agents.

Error recovery

  • Retry with backoff: If an action fails, wait and try again. Handles transient errors. Use exponential backoff with jitter to avoid thundering herds.
  • Fallback chains: Try alternative approaches when the primary fails. If the preferred model returns an error, try a different model.
  • Circuit breakers: After a threshold of failures, the circuit opens and all subsequent requests fail immediately without attempting the call. After a cooldown, allow a single test request. Borrowed directly from distributed systems engineering.

Context window management

As an agent session progresses, the context window fills up. Three strategies: Sliding window (keep only the most recent N turns — simple but lossy), Summarization (every 10 turns, generate a summary of the previous 10 and replace them — preserves key information but is non-deterministic), Selective retention (keep specific types of context and discard others — most targeted but requires understanding which context types matter).

Termination conditions

An agent without clear termination conditions will run until it hits a cost limit or timeout — neither of which is a good outcome. Termination conditions should be multi-layered: task completion (primary), cost limits (safety net), iteration limits (typically 50–100), time limits, and stall detection (same tool call 3 times without progress). Define "complete" explicitly for every task type.

Human-in-the-loop gates

Not every decision should be automated. Explicit gates where the agent pauses for human approval should be placed at points of high consequence: before modifying production data, before making irreversible changes, before spending above a cost threshold. The design of these gates matters — a gate that interrupts every five minutes is worse than no agent at all. A gate that only fires for genuinely consequential decisions adds safety without destroying flow.

When to use multiple agents

ScenarioWhy Multiple Agents

Fleet-scale parallelism

Some tasks aren't complex — they're repetitive. Migrating 200 microservices from Java 8 to Java 17. Patching a CVE across every repository. Updating documentation org-wide. These tasks need fleet-scale parallelism: hundreds of identical agents running the same playbook against different targets simultaneously. Work that would take a team weeks finishes in hours. The key requirement: isolation — each agent must run in its own environment so a failure in one repository doesn't affect others.

Multi-agent architectures

Hub-and-spoke: A central orchestrator delegates to specialist agents, collects results, and synthesizes a final output. Most common pattern — easiest to reason about and debug. The orchestrator is the single point of control. Downside: the orchestrator becomes a bottleneck and single point of failure.

Mesh: Agents communicate directly with each other, without a central coordinator. More resilient — no single point of failure — but harder to debug.

Pipeline: The output of one agent becomes the input of the next. Simple and predictable, but slow (each stage must complete before the next begins) and a failure at any stage blocks the entire pipeline.

When NOT to use multi-agent systems

Multi-agent systems add complexity. Don't use them when: the task is linear and doesn't require different expertise at different stages; the coordination cost exceeds the parallelism benefit; a single agent with better context could handle it. Do use them when: different stages require genuinely different expertise or permissions; tasks can genuinely be parallelized (the key word is "independent").

The observability challenge

The solution is distributed tracing with trace ID propagation. Every agent session gets a unique trace ID. When one agent delegates to another, the trace ID propagates to the child session. This creates a tree of traces showing the complete execution flow. Without trace ID propagation, debugging multi-agent failures is like debugging microservices without distributed tracing.

Ch 19 Agent Memory & Checkpoints +

The memory problem

Agents are stateless by default. Each LLM call is independent — the model has no memory of previous sessions. A team of ten engineers, each running five agent sessions per day, generates fifty sessions of institutional knowledge daily — and throws all of it away.

Types of agent memory

Session memory: The conversation history within a single agent run. Handled automatically by the agent framework. The challenge: it grows with every step, consuming context window space.

Persistent memory: Survives across sessions. When an agent finishes a task, it saves a summary of what it did, what it learned, and what went wrong. The next session can load these summaries to avoid repeating mistakes. The simplest implementation is an AGENTS.md file that agents read at the start of every session.

Shared memory: Makes one agent's knowledge available to others. When a code review agent discovers that a particular module has a tricky initialization sequence, that knowledge should be available to the code generation agent working on the same module.

Memory architecture decisions

File-based: Store session summaries as markdown files in the repository. Works for small teams but doesn't scale — summaries grow over time. Vector-based: Store memories as embeddings in a vector database (Chroma, Qdrant, Weaviate) and retrieve only the memories relevant to the current task. Scales better but retrieval quality is a concern. Graph-based: Store memories as nodes in a knowledge graph with typed relationships. Supports complex queries that vector retrieval can't handle.

For most teams: start with file-based memory (AGENTS.md plus session summaries), graduate to vector-based when the file-based approach stops scaling, and consider graph-based only when you need relationship-aware queries.

Learning from mistakes

The most valuable form of agent memory is learning from mistakes. When an agent produces output that a human corrects, that correction contains information about what the agent got wrong. The simplest implementation is a "lessons learned" file: "When modifying the payment service, always update the idempotency key. The agent has previously forgotten this, causing duplicate charges in testing."

Key insight: agent memory should be curated, not comprehensive. Storing every agent session produces large, noisy data. Storing only the corrections produces small, high-signal data that directly improves future performance.

Ch 20 AI Fatigue & Sustainable Adoption +

The fatigue pattern

In February 2026, a blog post titled "AI fatigue is real and nobody talks about it" hit #1 on Hacker News with 470+ points. An eight-month UC Berkeley Haas ethnographic study of a 200-person tech company found workers felt productive but were actually juggling: "a continual switching of attention, frequent checking of AI outputs, and a growing number of open tasks." Only 3% actual time savings despite feeling faster.

Five patterns from 300 engineers

Pattern 1: The meditative middle is gone. Coding has three psychological modes: problem-solving (stressful), coding (meditative flow state), seeing it work (dopamine). AI collapsed the meditative middle. Engineers now oscillate between stress and dopamine hits — the same pattern as doom-scrolling.

Pattern 2: Executive functioning fatigue. Pre-AI: [Hard decision] → [Implementation break] → [Hard decision]. Post-AI: [Hard decision] → [Hard decision] → [Hard decision] → [Burnout]. Without implementation as downtime, your prefrontal cortex runs at full load all day.

Pattern 3: The open source flood. PRs per week (maintainer) went from 20–25 to 100+. Sloppy/AI-generated PRs went from ~5% to ~40%. Total review time went from 5–6 hours/week to 25+ hours/week.

Pattern 4: Perceived cost aversion. AI generates a 70% solution in 1 minute. Spending 1 hour to improve it feels irrational. So you don't. And tech debt accumulates at AI speed.

Pattern 5: The slot machine. Variable reward schedule — sometimes it works perfectly, sometimes it doesn't — is the same mechanism that makes gambling addictive.

Sustainable adoption framework

Preserve thinking time. Block 2–3 hours per day for work without AI — architecture design, problem decomposition, code review. An engineer who can't think through a problem can't evaluate whether the AI's solution is correct.

Set explicit boundaries. Define which tasks use AI and which don't. Use AI for implementation, test generation, documentation, mechanical refactoring. Don't use AI for architecture decisions or security-critical code.

Manage the reward cycle. If the agent hasn't produced useful output in 15 minutes, stop and do it manually. Don't chase the "one more try" feeling.

Measure cognitive load, not just productivity. Track burnout indicators: overtime hours, sick days, turnover, self-reported stress levels.

The dependency problem

A colonoscopy study found that doctors using AI improved initially, but when AI was removed, performance dropped below their original baseline. The same pattern is emerging in software engineering — engineers who rely heavily on AI assistants report that whiteboard sessions become harder, their mental models of codebases atrophy, and their debugging skills decline. The engineers who handle this best deliberately practice without AI: morning thinking sessions, whiteboard-first design, regular "no-AI days."

Ch 21 The Conductor Model +

Engineers as orchestrators

The traditional engineering model: engineers write code. The emerging model: engineers orchestrate agents that write code. The analogy to a musical conductor is deliberate. A conductor doesn't play every instrument — they understand what each instrument can do, set the tempo, make judgment calls, and ensure the ensemble produces something coherent.

The conductor model works because it aligns human strengths with agent strengths. Humans are good at judgment, creativity, context understanding, and strategic thinking. Agents are good at execution, consistency, speed, and tireless repetition.

What conductors do (time allocation)

Writing effective task specifications

An effective task specification has five components: Goal (what should be different when the task is complete?), Context (what does the agent need to know?), Constraints (what should the agent NOT do?), Verification (how will we know the task is complete?), Scope (what files should the agent touch?). The specification doesn't need to be long — but it needs to be specific. Each sentence eliminates ambiguity and reduces the chance of the agent going off track.

The daily rhythm of a conductor

  • Morning (8–10am): Review overnight agent work — PRs, long-running task results, observability alerts. Highest leverage time.
  • Mid-morning (10am–noon): Specification and delegation. Write task specs, delegate to agents, start them running, move on.
  • Afternoon (1–3pm): Deep work — architecture design, design discussions, mentoring, complex code review. Protect this time aggressively. Don't check agent status.
  • Late afternoon (3–5pm): Review agent output, approve or re-delegate, update AGENTS.md, plan tomorrow's work.

Ch 23 Your First Agent in Production +

Pre-flight checklist

The overthinking camp spends months building infrastructure before running a single agent task. The underthinking camp gives an agent full access to their codebase on day one with no guardrails. Both approaches fail. The right approach is a controlled first deployment with clear boundaries, monitoring, and a plan for iteration.

Step 1: Prepare the repository (30 minutes)

Create an AGENTS.md with your project overview, setup commands, code conventions, testing requirements, and security notes (see Chapter 13 for the full template). Then create the agent configuration file at .agent/config.yaml with: model (e.g., claude-sonnet-4.6), max tokens per session (200,000), max cost per session ($3.00), timeout (30 minutes), filesystem/network/command permissions, and observability settings (tracing: true, cost_tracking: true, audit_logging: true).

Step 2: Configure MCP servers (30 minutes)

Connect the agent to the tools it needs: at minimum, filesystem (read and write code), version control (create branches and commits), and test runner (execute tests and read results). Start with 3–5 MCP servers. More than 10 creates tool definition bloat that wastes tokens on every turn. Verify each MCP server works independently before connecting it to the agent.

Step 3: Set up observability (45 minutes)

You cannot debug agent failures without traces. At minimum, every trace should capture: cost (cumulative per session), tool calls (name, arguments, response size, success/failure), LLM calls (model name, input/output token counts, latency, cost), session metadata (session ID, task description, initiating user). Set up two alerts from day one: sessions exceeding your cost cap ($3 default) and sessions exceeding your duration limit (30 minutes default).

Step 4: Configure CI/CD integration (30 minutes)

Agent-generated code must pass the same quality gates as human-written code. Configure your CI pipeline to run the full backpressure stack on every agent-created pull request: security scanning (Semgrep or CodeQL), tests (affected tests, not the full suite), linting, and type checking. Configure the agent to read CI failure output so it can self-correct. Set a maximum of 3 CI retry cycles.

Step 5: First run (30 minutes)

Pick a low-risk task: a well-defined bug fix with a failing test, a documentation update, or adding unit tests for an untested function. Avoid database migrations, authentication changes, or public API modifications. Write a clear task specification. Run the agent and watch the trace in real-time. Note where the agent hesitates, makes wrong assumptions, or lacks context — these observations will inform your AGENTS.md updates.

Ch 24 Security Checklist & Review Policies +

The 20-point security checklist

Before deploying any agent to production, verify twenty things across five categories. This checklist isn't aspirational — it's a gate. Score out of 20: below 12, don't deploy; 12–16, deploy with close monitoring; above 16, production-ready.

Authentication and Identity (4 pts): Unique agent identity (not shared service account), unique session ID in all traces, no hardcoded credentials in configs or prompts, API keys rotated on a defined schedule.

Authorization (4 pts): Least-privilege permissions, file access restricted to specific directories, network access limited to allowlisted domains, production database access requires explicit human approval per session.

Runtime Protection (4 pts): Agents run in sandboxed environments, cost budgets enforced with automatic termination, kill switch exists AND has been tested (not just configured), resource limits configured and enforced.

Input/Output Security (4 pts): No secrets in git history, .env files gitignored, tool responses validated against expected schemas, output filtering blocks exfiltration patterns.

Audit and Monitoring (4 pts): All agent actions logged with full traces, logs retained for at least 90 days, alerts fire on anomalous behavior, monthly access pattern reviews scheduled and conducted.

Two-layer review policy

Layer 1 (automated): Type checking, test execution, linting, security scanning, and architecture enforcement. Should run in under 2 minutes. Code that passes all automated checks proceeds to Layer 2. Invest in parallelizing checks and running only affected tests.

Layer 2 (human): Focused on judgment calls — architectural fit, business logic correctness, security implications, whether the approach makes sense in broader context. Give reviewers a structured checklist rather than asking them to review open-endedly. Structured review questions produce more consistent, higher-quality reviews than "looks good to me" approvals.

Access control policies: Four permission levels

  • Read-only agents: Can analyze code and generate documentation but cannot write files or execute commands. Default for new agent deployments.
  • Development agents: Can read and write source code and run build/test commands but cannot access production systems or secrets. Standard permission level for coding agents.
  • CI/CD agents: Can create pull requests, trigger pipelines, run deployment scripts but cannot merge without human approval.
  • Operations agents: Can access staging environments and read production logs but require human approval for all write actions. Grant sparingly.

Ch 25 Measuring Agent Impact +

Metrics that matter

Effectiveness metrics: Task completion rate (% of agent tasks accepted without major rework), first-attempt success rate (% that pass all automated checks on the first try), human override rate (% requiring significant human intervention).

Efficiency metrics: Average cycle time from task assignment to merged PR, average human review time per agent PR, cost per completed task.

Quality metrics: Defect rate in agent-generated code vs. human-generated code, regression rate (previously passing tests that now fail), post-merge bug rate.

Metrics that don't matter: Lines of code generated (more isn't better), number of PRs created (volume without quality is noise), raw token consumption (cost per task is what matters, not total tokens).

The metrics that mislead

"Time saved" is the most dangerous misleading metric. Teams often estimate time saved by comparing how long a task would have taken manually vs. how long it took with an agent — but don't account for the time spent specifying the task, reviewing the output, fixing issues, and managing the agent. A task that "would have taken 2 hours manually" and "took 15 minutes with an agent" might actually have taken 45 minutes total when you include the full workflow.

Connecting to business outcomes

The ultimate measure of agent impact isn't engineering metrics — it's business outcomes: time-to-market (are we shipping features faster?), quality (are we shipping fewer bugs?), cost efficiency (more output per engineering dollar?), and developer satisfaction (are our engineers happier and more productive?). Measure these alongside engineering metrics and look for correlations over time.

Why agent testing is different

Traditional software testing relies on deterministic inputs and outputs. The same prompt can produce different tool call sequences, different reasoning chains, and different final answers — all of which might be correct. The solution is to test at multiple levels:

UnitIndividual tool calls, prompt templatesDeterministic assertions
IntegrationTool chains, API interactionsMock services + assertions
BehavioralEnd-to-end task completionLLM-as-judge + golden datasets
RegressionPrompt changes don't break thingsBefore/after eval suites
PerformanceLatency, cost, token usageBenchmarks + budgets

The three testing patterns

Pattern 1: Deterministic assertions. For things you can check without an LLM: does the agent use the correct tools, does it respect token budgets, does it handle tool failures gracefully? These tests are fast (seconds) and catch obvious regressions.

Pattern 2: LLM-as-judge. Use a language model to evaluate the quality of another model's output. The judge model receives the task description, the agent's output, and a rubric, and produces a quality score. Key challenges: calibration (the judge has its own biases) and using a different model for judging than for generation. A well-calibrated judge achieves 85–90% agreement with human evaluators.

Ch 27 Enterprise Adoption & Vendor Strategy +

The $50M problem

A domain architect at a major global bank: "We allocated $50 million for AI initiatives. We're only allowed to use GitHub Copilot and Azure OpenAI. Our 300,000 token monthly limit gets exhausted by the 7th of each month. Teams are sharing tokens like rations." This is the norm, not an outlier.

The root cause: enterprise procurement processes were designed for SaaS subscriptions, not consumption-based AI services. Enterprises that budget for AI like they budget for SaaS consistently underestimate consumption by 3–5x.

Avoiding vendor lock-in

Model abstraction: Use LiteLLM or OpenRouter to abstract away model providers. Your business logic should work with any provider through an abstraction layer — if you're hardcoded to a single provider's API, you can't fall back.

Framework abstraction: Keep your framework integration thin. Your business logic should be framework-independent. If you can swap frameworks by changing the plumbing layer without touching the logic layer, you're not locked in.

The enterprise adoption playbook

  1. Start with a single, measurable use case — not "AI for everything." Pick one workflow where you can measure before/after.
  2. Build the abstraction layer first — set up LiteLLM or equivalent before writing any agent logic. This takes a day and saves months later.
  3. Instrument everything from day one — every token, every tool call, every latency measurement. You'll need this data to justify continued investment.
  4. Define success criteria before launch — "The agent should reduce code review turnaround from 48 hours to 4 hours for routine PRs, with a human override rate below 15%."
  5. Run shadow mode first — agent runs alongside humans, outputs are compared but not acted upon. Builds trust and catches issues before they matter.
  6. Gradual rollout with kill switches — start with one team, one repo, one workflow. Expand only after proving value.

Build vs. buy decision

Buy (managed platform) when: you want fast deployment, you don't have dedicated AI infrastructure engineers, you need enterprise features (SSO, audit logging, compliance) out of the box.

Build on a framework when: you have specific workflow requirements managed platforms don't support, you have AI infrastructure engineers to maintain the system, you need deep integration with internal systems.

Build custom when: you have unique requirements no framework supports, you need to run models on-premises for data sovereignty, you're building agent infrastructure as a core competency.

For most enterprises: start with a managed platform for the first use case, evaluate whether it meets your needs, and build custom only for the specific requirements the platform can't handle.

Ch 28 Agent Cost Control & FinOps +

Why agents are expensive by nature

A chatbot makes one LLM call per user message. An agent makes 3–10x more. A single user request can trigger planning, tool selection, execution, verification, and response generation. And unlike chatbot costs (relatively predictable), agent costs are highly variable — a simple bug fix might take 3 tool calls and cost $0.05; a complex refactoring might take 50 tool calls and cost $5.00; a runaway agent stuck in a retry loop might cost $15.00 before anyone notices.

The four pillars of agent cost control

Pillar 1: Model routing. Use the cheapest model that can handle the task. A test generation task routed to GPT-5.2-mini at $0.15/$0.60 costs 95% less than the same task routed to Opus 4.6 at $5/$25 — and the output quality is comparable. Research shows organizations using model routing save 40–85% compared to single-model approaches.

Pillar 2: Caching. Cache at multiple levels — prompt caching (supported by Anthropic and OpenAI for repeated system prompts), semantic caching (cache responses for semantically similar queries), and tool result caching (cache expensive tool calls like database queries).

Pillar 3: Token budgets. Set hard limits per task: default $3 for routine tasks, $10 for complex tasks. Kill the agent automatically when the limit is exceeded.

Pillar 4: Prompt optimization. Remove redundant instructions (10–30% savings), use structured examples instead of verbose descriptions (20–40% savings), compress system prompts (15–25% savings), prune conversation history (40–70% savings — but risk losing context).

Building a cost culture

Cost control is a cultural problem as much as a technical one. Three practices build a cost-conscious culture: make costs visible (every engineer should see their daily spend in a dashboard — personal spend, not just team spend); set budgets with consequences (a budget without consequences is a suggestion); celebrate cost optimization (when an engineer discovers that routing test generation to a cheaper model saves $500/month with no quality loss, share that win).

Ch 29 Agent Governance & Compliance +

Why governance can't be an afterthought

In February 2026, UC Berkeley's Center for Long-Term Cybersecurity published the "Agentic AI Risk-Management Standards Profile" — the first formal framework for managing risks specific to agentic AI. The four risk categories: unauthorized actions (the agent does something it shouldn't), cascading failures (one agent's error propagates through a multi-agent system), opacity (inability to explain why the agent took a specific action), and emergent behavior (agents developing unexpected patterns).

The agent audit trail

An effective agent audit trail captures five categories: Actions (every tool call, file modification, API request, and command execution with timestamps and results), Decisions (the model's reasoning at each step), Authorization (every permission check, what was allowed and denied), Data access (every piece of data the agent read or wrote, classified by sensitivity), Cost (token consumption and dollar cost per action). The audit trail must be immutable, searchable, and retained for at least 90 days.

Compliance frameworks applied to agents

SOC 2: Requires authorization controls (OpenFGA), real-time monitoring, prompt versioning, and PII detection/redaction. GDPR: Agents should access only necessary data; users can request human review of agent decisions; DPIA required for high-risk agent deployments. EU AI Act (effective 2026): Most enterprise agents fall into "limited risk" (transparency: users must know it's AI) or "high risk" (conformity assessment, human oversight, logging required).

Governance checklist

  • Kill switch exists to disable the agent immediately
  • Audit logging captures all agent actions
  • Agent has a defined scope and permission boundary
  • PII detection/redaction is in the pipeline
  • Data classification is applied to all accessed resources
  • Human escalation paths are defined for high-risk actions
  • Token/cost budgets are enforced
  • Incident response plan includes agent-specific scenarios
  • Compliance mapping is documented (SOC 2, GDPR, etc.)
  • Regular access reviews include agent permissions

Ch 30 Structured Outputs & Function Calling +

The reliability problem

Agents need to produce structured data — JSON for tool calls, typed parameters for function invocations. A model that produces valid JSON 95% of the time sounds reliable — until you realize that in a 20-step agent session, the probability of at least one invalid output is 1 - 0.95^20 = 64%. You need 99.9% or better for production agent systems.

Instructor is the most practical tool for production structured outputs. It wraps any LLM provider with Pydantic validation and automatic retries. Schema design principles

  • Keep schemas flat — deeply nested objects confuse models

  • Use enums for fixed choices
  • Name tools with verbs: read_file, search_code, not file, search

Ch 31 Model Selection & Routing +

The end of "pick one model"

February 2026 saw GPT-5.3-Codex, Claude Opus 4.6 and Sonnet 4.6 with 1M context, and Gemini 3.1 Pro all ship within fourteen days. No single model dominates across all dimensions: GPT-5.3-Codex leads on Terminal-Bench (75.1%), Claude Opus 4.6 leads on SWE-bench Verified (80.8%), Gemini 3.1 Pro leads on long-context tasks. Routing architectures

Rule-based routing: Define rules based on task type ("code_generation" → sonnet-4.6, "summarization" → gpt-5.2-mini). Simple, auditable, cheap. Best starting point.

Cascade routing: Try the cheapest model first; escalate to more expensive models only if the output doesn't meet quality criteria. Research from ETH Zurich shows cascade routing reduces costs by 40–60% compared to always using the best model, with less than 2% quality degradation. 60–70% of tasks are simple enough for cheap models.

Semantic routing: Use embeddings to match tasks to specialized model configurations. Handles novel task types gracefully but is less predictable than rule-based routing.

The model selection decision tree

  1. Does the task require reasoning across multiple files or systems? → If yes, use a frontier model (Opus 4.6, GPT-5.2, Gemini 3.1 Pro)
  2. Does the task require code generation or modification? → If yes, use a mid-tier coding model (Sonnet 4.6 or GPT-5.2)
  3. Is the task primarily text processing (summarization, classification, extraction)? → If yes, use a lightweight model (DeepSeek V3.2, Gemini 3 Flash)
  4. Does the task involve sensitive data that can't leave your infrastructure? → If yes, use a self-hosted open-source model regardless of task complexity

Ch 32 Backpressure & Automated Feedback +

The core problem

AI made code generation fast. It didn't make code verification fast. The human became the bottleneck. An engineering lead at an automotive SaaS company: one engineer reviewed 130 pull requests in 15 days — about 20 file changes each. 20–30% of the AI-generated code required manual fixes.

Backpressure for agents means automated feedback loops that catch errors before they reach a human. Every error caught by a type checker, test suite, or linter is one fewer error in your review queue.

The backpressure hierarchy (strongest to weakest)

LayerWhat It CatchesAgent Self-Correction Rate
Type system (strict TS, Rust, Go)Wrong types, missing fields, impossible states~95%
Test suite (fast, < 2 min)Regressions, logic errors~80%
Linters + pre-commit hooksStyle, dead code, imports~99%
Architecture enforcement (ArchUnit)Boundary violations, circular deps~90%
Visual verification (Playwright)UI regressions~70%
Human reviewJudgment calls, architecture fitN/A

Human review should be last, not first. By the time code reaches you, the trivial issues should already be resolved.

The complete backpressure pipeline

Stage 1: Static Analysis — Within seconds, linters, type checkers, and formatters run against the diff. Failures are returned as structured feedback (file, line number, rule violated, suggested fix). Catches 40–60% of issues in under 10 seconds.

Stage 2: Test Execution — Runs only affected tests (using jest --changedSince or pytest-testmon). Catches logic errors and regressions in 30–90 seconds.

Stage 3: Security Scanning — SAST tools (Semgrep, CodeQL, or Snyk Code) scan for vulnerabilities. Non-negotiable for production agents.

Stage 4: Architecture Enforcement — Custom rules verify that the change follows architectural conventions. Tools: ArchUnit (Java), Dependency Cruiser (JavaScript). Catches violations that type systems and tests miss.

Stage 5: Human Review Gate — Only code that passes all four automated stages reaches a human reviewer. Their job: evaluate what automation can't — architectural fit, business logic correctness, whether the approach makes sense.

Tuning backpressure

The target: full iteration cycle (edit → check → fix) in under 2 minutes. Over 5 minutes, you're losing too much to wait time. Under 30 seconds, check that your tests are catching real issues.

Too little backpressure: Hallucinations pass through. Add more automated checks. Strengthen test coverage.

Too much backpressure: Feedback loop is too slow. Parallelize checks. Run only affected tests. Cache build artifacts.

The ROI of engineering infrastructure in the agent era

A strict TypeScript configuration catches roughly 20 type errors per day in a typical codebase. Before agents, each error would have been caught by a developer during testing — maybe 5 minutes per error, or 100 minutes per day. With agents, each error is caught by the compiler and fed back to the agent, which self-corrects in seconds. The same 100 minutes saved, but now it's saving the developer from reviewing agent output rather than from debugging their own code. The agent era makes engineering infrastructure investments pay off faster.

Ch 33 The Agent Adoption Playbook +

Prerequisites

Before starting the adoption playbook, verify three prerequisites: (1) CI/CD pipeline running in under 5 minutes — agents need fast feedback to self-correct; (2) Test coverage of at least 60% for the areas where agents will work — tests are the primary backpressure mechanism; (3) Type system or linting — TypeScript strict mode, Rust, Go, or a strict linting configuration. If you don't meet these prerequisites, invest in them first.

Week-by-week implementation

Week 1: Foundation. Goal: establish automated feedback infrastructure before introducing agents. Audit your feedback loops, write AGENTS.md, enable strict tooling (TypeScript strict mode, ESLint strict rules, pre-commit hooks that run tsc --noEmit && eslint . && npm test).

Week 2: First agent tasks. Goal: run agents on low-risk tasks with full backpressure. Start with: writing unit tests for existing code, adding input validation, fixing lint warnings, updating documentation. Do NOT start with database migrations, authentication changes, public API modifications, or infrastructure changes.

Week 3: Measure and adjust. Goal: review Week 2 data and adjust backpressure. If acceptance rate is above 70%, expand to more task types. If 50–70%, investigate failures. If below 50%, fix the foundation first before expanding.

Week 4: Team expansion. Goal: move from one engineer to the full team. Run a 30-minute team session, set up a shared Slack channel for agent wins and failures, assign each team member one low-risk task to try with an agent.

Weeks 5–8: Scale. Expand task types, implement model routing, add multi-agent workflows, formalize the process. Document everything learned. Update AGENTS.md. Write a team playbook.

Common adoption failures

  • The Big Bang: Going all-in overnight with no infrastructure. Fix: follow the week-by-week playbook.
  • The Skeptic's Veto: One senior engineer resists, adoption stalls. Fix: don't require universal adoption; let data speak.
  • The Prompt Obsession: Spending weeks perfecting prompts instead of building infrastructure. Fix: a mediocre prompt with excellent context and strong backpressure outperforms a perfect prompt with no infrastructure.

  • The Measurement Gap: Not measuring impact. Fix: set up measurement from day one — log every task, track every cost, measure every outcome.
